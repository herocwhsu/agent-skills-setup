# Portable Skills Refactor — Design

**Date:** 2026-05-19
**Status:** Approved (pending spec review)
**Goal:** Make `agent-skills-setup` usable by anyone, not just `hero.hsu@vivotek.com`. Eliminate duplicated logic. Provide clean install/uninstall semantics.

---

## Problem Statement

Today the repo is hard-coded for one user:

- `vivotek.atlassian.net`, `confluence.vivotek.com`, `hero.hsu`, `hero.hsu@vivotek.com`, project key `VOR` appear inside three shipped `SKILL.md` files. Cloning the repo as anyone else produces broken behavior — agents will literally call those hosts.
- The slug-generation rule (`s|https\?://||;s|[^a-zA-Z0-9]|-|g;s/-\+/-/g;s/-$//`) is duplicated in 8+ places across credentials scripts and skill files. A single change requires editing all of them in lockstep, and they drift silently.
- The `html2md.py` discovery loop (4-directory probe across `~/.kiro/skills`, `~/.claude/skills`, `~/.copilot/skills`, `~/.codex/skills`) is duplicated 3 times across bash and Python, in two different SKILL.md files.
- `confluence.sh` and `jira.sh` are 95% identical with only the prompt label and slug prefix differing.
- `install.sh` and `install.ps1` have drifted — PS1 supports `-Agent kiro` (non-interactive), bash does not. Bash special-cases kiro prompts; PS1 doesn't.
- `uninstall_github_skill` re-downloads the GitHub zip just to learn skill names — uninstall requires network access.
- Kiro prompts are special-cased outside the registry, breaking the README's "registry is single source of truth" promise.

## Goals

1. Anyone cloning the repo can run install + setup-credentials and get working skills against their own Jira/Confluence.
2. Slug rule, html2md detection, and credential I/O each live in exactly one place.
3. Install/uninstall is offline-capable and idempotent.
4. Existing keychain entries migrate automatically — no manual re-entry.

## Non-Goals

- Adding a test framework. Manual smoke tests are sufficient for this refactor.
- Adding a Python helper module. Bash + the existing `_store.sh` is enough.
- Reworking the agent-skills directory layout (`~/.kiro/skills/`, etc.) — those stay per-agent.
- Touching `html2md.py` itself. It works.

## Architecture

### New cross-agent layer: `~/.agent-skills-setup/`

A single top-level dotdir holds two files that don't belong to any individual agent:

| File | Role |
|---|---|
| `~/.agent-skills-setup/config.sh` | Non-secret user values (hosts, usernames, project key) |
| `~/.agent-skills-setup/lib.sh` | Shared bash helpers callable from skills |

Naming follows the existing pattern (`~/.kiro/`, `~/.claude/`, `~/.copilot/`, `~/.codex/` are all top-level dotdirs). Matches the GitHub repo name exactly to minimize collision risk.

**Windows equivalent:** `%APPDATA%\agent-skills-setup\` for the same two files.

### Three-layer separation

```
~/.<agent>/skills/<name>/SKILL.md      ← per-agent, only the SKILL.md content
~/.agent-skills-setup/                  ← cross-agent: config + helpers (NEW)
keychain (agent-skills-setup: prefix)   ← cross-agent: secrets (renamed)
```

Skills load all three at runtime:

```sh
# Top of every skill that needs it
source ~/.agent-skills-setup/lib.sh
load_config   # sources ~/.agent-skills-setup/config.sh, errors with hint if missing
```

## Component Specifications

### config.sh

Generated by `setup-credentials.sh` as a side effect of the existing add/update prompts. Plain shell-sourceable:

```sh
# ~/.agent-skills-setup/config.sh — generated by setup-credentials.sh
# Edit by hand if needed; setup-credentials.sh re-reads on next run and pre-fills prompts.

JIRA_HOST="vivotek.atlassian.net"
JIRA_USER="hero.hsu@vivotek.com"
JIRA_PROJECT_KEY="VOR"
CONFLUENCE_HOST="confluence.vivotek.com"
CONFLUENCE_USER="hero.hsu"
APIDOG_USER="hero.hsu@vivotek.com"
```

Behavior:
- `setup-credentials.sh confluence add` writes `CONFLUENCE_HOST` + `CONFLUENCE_USER`.
- `setup-credentials.sh jira add` writes `JIRA_HOST` + `JIRA_USER` and additionally prompts for `JIRA_PROJECT_KEY`.
- `setup-credentials.sh apidog add` writes `APIDOG_USER`.
- If config.sh exists, prompts pre-fill from current values; press Enter to keep.
- Each `add` updates only the relevant subset; other keys are preserved.

### lib.sh API

All functions are bash; no Python. POSIX-where-possible, but bash 3+ features (declare -A, etc.) are acceptable since macOS ships bash 3.2.

```sh
# ---------- URL slug rules ----------
slugify_url <url>
  # In:  https://confluence.vivotek.com
  # Out: https---confluence-vivotek-com
  # macOS BSD sed and GNU sed both produce identical output.
  # Note: 'https://' is NOT stripped — preserves backward-compat with existing keychain slugs.

service_slug <prefix> <url>
  # In:  jira https://vivotek.atlassian.net
  # Out: jira-https---vivotek-atlassian-net

# ---------- html2md detection ----------
find_html2md
  # Echoes absolute path to html2md.py, or exits non-zero with stderr message.
  # Searches ~/.kiro/skills, ~/.claude/skills, ~/.copilot/skills, ~/.codex/skills.

# ---------- config + secrets ----------
load_config
  # Sources ~/.agent-skills-setup/config.sh.
  # If missing, prints a clear "run setup-credentials.sh" message and returns 1.

read_secret <slug> <user>
  # Wrapper around _store.sh read_credential. Echoes secret to stdout (never logs).
  # Returns 1 if not found.

require_secret <slug> <user> [hint]
  # read_secret + exits 1 with a "run scripts/credentials/<service>.sh add" hint.
```

Implementation notes:
- `lib.sh` sources `_store.sh` from inside the install dir, so credential I/O lives in one module. The skills installation copies `_store.sh` into `~/.agent-skills-setup/` alongside `lib.sh`.
- `slugify_url` deliberately keeps `https://` because existing keychain entries already include it (BSD sed quirk documented in the original SKILL.md). Removing it would break migration; users who don't care about migration are unaffected.

### Collapsed credential scripts

`scripts/credentials/service.sh` replaces `confluence.sh` + `jira.sh` + `apidog.sh`:

```sh
bash scripts/credentials/service.sh <service> <action>
  # services: confluence | jira | apidog
  # actions:  add | update | delete | list | verify
```

Service definitions live inline as a small table:

```sh
declare -A SERVICE_PROMPTS=(
  [confluence]="Confluence URL (e.g. https://confluence.example.com)"
  [jira]="Jira URL (e.g. https://your-org.atlassian.net)"
  [apidog]=""   # no URL — fixed slug 'apidog'
)
declare -A SERVICE_EXTRA=(
  [jira]="JIRA_PROJECT_KEY"   # extra config.sh key prompted for jira
)
```

Old `confluence.sh`/`jira.sh`/`apidog.sh` become thin shims that delegate to `service.sh` with the right argument. Shims kept for one release for backward-compat, then deleted.

### Skill rewrites

Three SKILL.md files change. The patterns:

**Before** (excerpt from `fetch-jira-story/SKILL.md`):
```sh
JIRA_HOST="vivotek.atlassian.net"
SLUG="jira-$(echo "https://$JIRA_HOST" | sed 's|https\?://||;s|[^a-zA-Z0-9]|-|g;s/-\+/-/g;s/-$//')"
_JIRA_USER="hero.hsu@vivotek.com"
_JIRA_PASS=$(security find-generic-password -s "agent-skills:$SLUG" -a "$_JIRA_USER" -w 2>/dev/null)
if [ -z "$_JIRA_PASS" ]; then
  echo "ERROR: Jira credential not found." >&2
  exit 1
fi
```

**After**:
```sh
source ~/.agent-skills-setup/lib.sh
load_config

SLUG=$(service_slug jira "https://$JIRA_HOST")
_JIRA_PASS=$(require_secret "$SLUG" "$JIRA_USER" "scripts/credentials/service.sh jira add")
```

`fetch-page-to-markdown/SKILL.md` and `create-story-tasks/SKILL.md` get the same treatment.

`html2md.py` invocation collapses from a 7-line probe + error check to:
```sh
HTML2MD=$(find_html2md) || exit 1
```

`plan-story/SKILL.md` and `fetch-page-to-markdown/html2md.py` are unchanged.

### Install/uninstall changes

**install.sh:**
1. New step: create `~/.agent-skills-setup/`, copy `lib.sh` and `_store.sh` into it (idempotent — `cp -f`).
2. New step: run keychain migration (see below).
3. New step: write `installed.txt` to `~/.agent-skills-setup/` listing all skill names installed this run, for offline uninstall.
4. Existing registry loop unchanged.
5. Keep the kiro-prompts copy block (lines 36–46) — prompts genuinely are kiro-specific (no other agent has a `prompts/` directory concept), so a special case is appropriate, not a smell. Move it into a named function `install_kiro_prompts()` in `_lib.sh` for clarity. install.ps1 gains the same function.
6. Add `--agent <name>` flag matching the existing PS1 behavior. `select_agents()` only prompts when no flag provided.

**install.ps1:**
- Mirror all the above. Already has `-Agent` flag — unchanged. Add lib.sh copy, migration call, installed.txt write.

**uninstall.sh / uninstall_github_skill:**
- Read `~/.agent-skills-setup/installed.txt` to know which skill dirs to remove. No network access required.
- Fall back to current behavior (re-download zip) only if `installed.txt` is missing.

### Keychain migration

One-shot, idempotent, runs every time install.sh executes. Removable in a future release once all installs have migrated.

```sh
migrate_keychain() {
  local count=0
  case "$(_os)" in
    darwin)
      while IFS= read -r svc; do
        # svc: agent-skills:jira-https---vivotek-atlassian-net
        local newsvc="agent-skills-setup:${svc#agent-skills:}"
        local user pass
        user=$(extract_account_for_service "$svc")
        pass=$(security find-generic-password -s "$svc" -a "$user" -w 2>/dev/null) || continue
        security add-generic-password -s "$newsvc" -a "$user" -w "$pass" 2>/dev/null
        security delete-generic-password -s "$svc" -a "$user" 2>/dev/null
        count=$((count+1))
      done < <(security dump-keychain 2>/dev/null | python3 -c "
import sys, re
for entry in sys.stdin.read().split('keychain:'):
    m = re.search(r'\"svce\"<blob>=\"(agent-skills:[^\"]+)\"', entry)
    if m: print(m.group(1))
")
      ;;
    linux-gui|windows)
      # equivalent loops via secret-tool / cmdkey
      ;;
  esac
  if [[ $count -gt 0 ]]; then
    echo "  ✓ migrated $count keychain entries to agent-skills-setup: prefix"
  fi
}
```

`_store.sh` `_KEYCHAIN_PREFIX` constant changes from `agent-skills` to `agent-skills-setup`.

## Data Flow Examples

### A skill running fetch-jira-story

```
1. Agent invokes: bash <SKILL.md commands>
2. SKILL.md sources ~/.agent-skills-setup/lib.sh
3. lib.sh sources _store.sh (sibling)
4. SKILL.md calls load_config → sources ~/.agent-skills-setup/config.sh
   → JIRA_HOST, JIRA_USER, JIRA_PROJECT_KEY now in env
5. SKILL.md calls service_slug jira "https://$JIRA_HOST"
   → returns "jira-https---vivotek-atlassian-net"
6. SKILL.md calls require_secret "$SLUG" "$JIRA_USER"
   → reads from agent-skills-setup:jira-https---vivotek-atlassian-net
7. curl -u "$JIRA_USER:$_PASS" → fetch story
8. Python block parses, writes ./docs/stories/<ID>/story.md
```

No hardcoded values. Same flow works for any user with their own config.sh.

### Fresh install on a new machine

```
1. git clone … && bash scripts/install.sh
2. install.sh creates ~/.agent-skills-setup/{lib.sh,_store.sh}
3. install.sh runs migration (no-op, nothing to migrate)
4. install.sh installs skills per registry
5. install.sh writes ~/.agent-skills-setup/installed.txt
6. User runs: bash scripts/setup-credentials.sh jira add
7. service.sh prompts: Jira URL, Username, API token, JIRA_PROJECT_KEY
8. service.sh writes config.sh + keychain entry under agent-skills-setup:
9. Subsequent skill runs work.
```

### Existing install upgrading

```
1. cd agent-skills-setup && git pull && bash scripts/install.sh
2. install.sh creates ~/.agent-skills-setup/ if missing
3. Migration loop finds N agent-skills:* entries, copies to agent-skills-setup:*, deletes old
4. Reinstall skills (now reference new prefix)
5. config.sh does not yet exist → first skill invocation prints
   "ERROR: ~/.agent-skills-setup/config.sh missing. Run: bash scripts/setup-credentials.sh jira add"
6. User runs that. Done.
```

The one user-visible upgrade step is running setup-credentials.sh once per service to populate config.sh. Migration handles the secrets automatically.

## Error Handling

- **config.sh missing:** `load_config` prints a one-line hint and returns 1. Skill aborts with clear message.
- **Required key not in config.sh:** Each skill checks `[[ -z "${JIRA_HOST:-}" ]]` after `load_config` and prints "missing JIRA_HOST in config.sh — run setup-credentials.sh jira add".
- **Keychain entry missing:** `require_secret` prints "credential not found at <slug> for <user>. Run: <hint>" and exits 1.
- **html2md.py missing:** `find_html2md` prints "html2md.py not found in any agent skills directory. Run install.sh." and returns 1.
- **lib.sh missing:** Skill's `source` call fails with bash's default error, falling through to the user. Documented as "if you see 'No such file or directory' for lib.sh, run install.sh".

## Backward Compatibility

| Concern | Resolution |
|---|---|
| Existing `agent-skills:*` keychain entries | Auto-migrated on first install.sh run after upgrade |
| Old `confluence.sh`/`jira.sh`/`apidog.sh` paths | Kept as shim wrappers for one release, then removed |
| User has hardcoded paths in their own scripts | None — those scripts must update; documented in CHANGELOG |
| Skills installed before refactor | `install.sh` reinstalls them with new SKILL.md content |

## Verification Plan

Manual smoke tests after refactor (no test framework introduced):

1. **Fresh install:**
   - Clone in a clean dir, `bash scripts/install.sh --agent claude`
   - Confirm `~/.agent-skills-setup/{lib.sh,_store.sh,installed.txt}` exist
   - Confirm `~/.claude/skills/` populated
   - `bash scripts/setup-credentials.sh jira add` → confirm config.sh written, keychain entry under `agent-skills-setup:`

2. **Upgrade path:**
   - Manually create `agent-skills:jira-test` keychain entry to simulate old install
   - Run `bash scripts/install.sh`
   - Confirm entry now lives at `agent-skills-setup:jira-test`, old one gone
   - Re-run install.sh — confirm migration log shows 0 entries (idempotent)

3. **End-to-end skill exec:**
   - With config.sh + creds in place, ask the agent to run fetch-jira-story against a real ticket
   - Confirm `./docs/stories/<ID>/story.md` written

4. **Offline uninstall:**
   - Disconnect network
   - `bash scripts/uninstall.sh --agent claude`
   - Confirm skills removed without network errors

5. **Cross-agent:**
   - `bash scripts/install.sh --agent all`
   - Run any skill from any of the 4 skill dirs — all should work because all read the same `~/.agent-skills-setup/` files

If automated tests are wanted later, bats-core is the natural fit. Out of scope for this refactor.

## Risks

- **Migration data loss:** if `security delete-generic-password` runs after a partial-failure `security add-generic-password`, an entry could be lost. Mitigation: migration always reads → writes new → only deletes old if new write succeeded.
- **Cross-platform parity drift:** install.sh and install.ps1 must stay in sync. Mitigation: both scripts call out to the same registry.txt; differences are limited to OS-specific syscalls.
- **`installed.txt` getting stale:** if user manually deletes a skill dir, uninstall would try to remove a non-existent path. Mitigation: `remove_skill` already tolerates missing paths.
- **install.ps1 missing kiro prompts:** install.ps1 currently doesn't copy `prompts/*.md` to `~/.kiro/prompts/` at all (bash-only behavior today). Refactor adds parity by introducing `Install-KiroPrompts` in install.ps1. Verify kiro prompts land in `~/.kiro/prompts/` on both platforms after refactor.
